# 第二阶段：权限和留痕设计

## 目标

第二阶段完成员工敏感数据隔离和统一审计能力，为第三阶段的每日交账、退回、确认和更正提供可复用的留痕基础。

本阶段不实现每日交账、换现和旧料出库。第一阶段已经完成的店铺与员工管理继续保留，本阶段为这些管理操作补充审计记录。

## 范围

本阶段包含：

- 员工不能通过页面或接口查看、筛选、创建店铺支出；
- 员工继续只能访问所属店铺的数据；
- 老板修改旧流水时必须填写原因；
- 流水删除改为作废，原始记录永久保留；
- 作废流水不参与资金、库存和统计计算；
- 流水修改、流水作废、店铺修改、员工修改和员工停用均写入审计日志；
- 老板可以查看和筛选审计日志。

本阶段不包含：

- 每日交账单及其提交、确认、退回；
- 换现双边记账；
- 旧料出库、成本和利润；
- 员工旧料出库授权；
- 首页异常提醒。

## 权限设计

### 老板

老板可以：

- 查看所有店铺或指定店铺的业务流水；
- 查看和创建店铺支出；
- 修改旧流水，但必须填写修改原因；
- 作废流水，但必须填写作废原因；
- 查看全部审计日志，并按店铺、对象、动作和日期筛选；
- 修改店铺和员工资料、停用或启用员工。

老板在“全部店铺”上下文中只能读取，修改和作废操作直接使用目标记录原有的店铺归属，不允许借请求头改变记录所属店铺。

### 员工

员工继续由服务器强制绑定所属店铺。无论请求头或查询参数如何传递，都不能访问其他店铺。

员工不能：

- 查询任何 `operating_expense` 流水；
- 使用业务类型筛选绕过支出隔离；
- 创建 `operating_expense` 流水；
- 修改或作废已保存流水；
- 查看审计日志。

即使员工拥有 `transactions` 权限，以上限制仍然成立。回收专员继续只能访问已授权的回收类型。

员工仪表盘统计不得返回店铺支出明细或支出金额。员工需要的销售、回收和资金数据继续按现有权限提供。

## 审计模型

新增 `audit_logs` 表，字段包括：

- `id`；
- `store_id`，允许为空，用于不属于单店的老板账户操作；
- `actor_admin_id`，执行操作的后台账户；
- `subject_type`，被操作对象类型；
- `subject_id`，被操作对象编号；
- `action`，动作名称；
- `reason`，修改或作废原因；
- `before_data`，修改前 JSON；
- `after_data`，修改后 JSON；
- `created_at`。

动作名称首批固定为：

- `transaction.updated`；
- `transaction.voided`；
- `store.updated`；
- `user.updated`；
- `user.disabled`；
- `user.enabled`。

`before_data` 和 `after_data` 保存业务字段快照，不保存密码、API token 等秘密字段。员工密码重置不记录密码内容。

新增 `AuditLog` 模型和 `AuditLogger` 服务。控制器通过该服务写日志，避免各控制器自行拼接日志结构。写日志和业务修改必须处于同一个数据库事务中，任一操作失败时全部回滚。

审计日志只允许新增和查询，不提供修改、删除或作废接口。

## 流水作废

在 `transactions` 表新增：

- `voided_at`；
- `voided_by_admin_id`；
- `void_reason`。

作废操作要求老板提交非空原因。系统在数据库事务中：

1. 验证流水尚未作废；
2. 保存作废前快照；
3. 写入作废时间、操作人和原因；
4. 创建 `transaction.voided` 审计日志。

重复作废返回 `422`，不产生第二条日志。

所有流水列表、资金余额、库存、月统计和账户明细默认排除已作废记录。老板流水列表可以通过显式筛选查看“有效”“已作废”或“全部”，已作废行清楚显示状态和原因。

作废流水不能再次修改。恢复作废不在本阶段范围内；需要纠正时由老板新建正确流水并保留原作废记录。

## 流水修改

老板修改流水必须提交 `change_reason`，至少 2 个字符，最多 500 个字符。`change_reason` 不写入流水业务字段。

修改过程在数据库事务中：

1. 拒绝修改已作废流水；
2. 读取并保存修改前快照；
3. 使用现有校验和归一化逻辑更新流水；
4. 保存修改后快照；
5. 创建 `transaction.updated` 审计日志。

流水所属店铺、原始录入人和后台操作人不能通过修改接口改变。

## 店铺与员工留痕

店铺资料修改后记录 `store.updated`。员工资料、所属店铺和权限修改记录 `user.updated`。

员工启用状态发生变化时，除 `user.updated` 外，使用更具体的 `user.disabled` 或 `user.enabled` 动作，避免同一次请求产生两条重复日志；启用状态变化时只写具体动作，其他修改写 `user.updated`。

审计快照排除：

- `password`；
- `api_token`；
- 其他认证秘密。

店铺和员工管理沿用现有老板专属权限，不改变第一阶段接口的主要请求格式。

## 接口

### 修改流水

`PUT /api/admin/transactions/{transaction}`

在现有业务字段之外要求：

```json
{
  "change_reason": "更正录入金额"
}
```

### 作废流水

`DELETE /api/admin/transactions/{transaction}`

请求体：

```json
{
  "reason": "重复录入"
}
```

成功返回作废后的流水状态，不再返回 `deleted`。

### 查询审计日志

`GET /api/admin/audit-logs`

支持：

- `store_id`；
- `subject_type`；
- `action`；
- `date_from`；
- `date_to`；
- `page`；
- `per_page`。

响应包含操作人、店铺、动作、原因、修改前后内容和操作时间。该接口仅老板可访问。

## 前端

流水编辑对话框增加必填“修改原因”。删除按钮和提示文字改为“作废”，作废确认框要求填写原因。

流水表增加状态筛选和状态列：

- 有效；
- 已作废。

已作废流水使用明确但克制的状态样式，禁用编辑和再次作废操作。

新增老板专属“操作记录”页面，使用表格显示：

- 时间；
- 操作人；
- 店铺；
- 操作类型；
- 对象；
- 原因；
- 详情。

详情使用抽屉展示修改前和修改后字段。员工导航中不显示该入口，但安全边界仍由后端接口执行。

员工界面隐藏支出按钮、支出筛选项、支出统计和支出记录。前端隐藏仅用于改善体验，后端必须独立拒绝相关请求。

## 错误处理

- 缺少修改或作废原因时返回 `422`；
- 员工访问支出或审计日志时返回 `403`；
- 修改或作废其他店铺记录的员工请求返回 `403`；
- 修改已作废流水或重复作废时返回 `422`；
- 目标记录不存在时返回 `404`；
- 审计日志写入失败时，业务修改同步回滚。

错误响应继续使用现有 JSON `message` 格式。

## 测试

后端功能测试覆盖：

- 普通流水员工不能读取、筛选或创建支出；
- 回收专员不能读取支出；
- 员工统计响应不包含支出金额；
- 老板仍可查看和创建支出；
- 修改流水缺少原因时失败；
- 修改流水生成包含前后快照、原因、操作人和店铺的日志；
- 作废流水保留原记录并生成日志；
- 作废流水从余额、库存、月统计和账户明细中排除；
- 重复作废和修改已作废流水失败；
- 店铺和员工修改生成无秘密字段的日志；
- 员工不能访问审计日志；
- 老板可以按店铺和动作筛选审计日志。

前端验证包括：

- 生产构建成功；
- 老板可以填写原因修改和作废流水；
- 作废后统计刷新且流水显示已作废；
- 老板可以查看审计详情；
- 员工界面没有支出及操作记录入口；
- 使用 Google Chrome 验证主要流程和响应式布局。

## 验收标准

- 员工无法通过页面或接口获得店铺支出金额和明细；
- 员工无法通过伪造店铺编号访问其他店铺；
- 老板修改和作废旧流水必须填写原因；
- 作废记录永久保留，但不影响当前资金、库存和统计；
- 老板能看到修改前、修改后、操作人、店铺、时间和原因；
- 店铺和员工资料变更留下审计记录；
- 审计记录不可通过接口修改或删除；
- 第一阶段的多店铺、账户和历史数据结果不回退；
- 第三阶段可以复用 `AuditLogger` 记录提交、退回和确认。
